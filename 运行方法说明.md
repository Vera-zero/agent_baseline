# ETE-Graph Agent 方法运行指南

本文档介绍如何运行 `/workspace/ETE-Graph/agent` 目录中的四种时序问答方法，统一使用 **BGE-M3** 作为 embedding 模型。

## 目录结构

```
agent/
├── MRAG-master/           # MRAG 方法原代码
├── timeqa_baseline_lab/   # 包含 zero_shot_cot、rag_cot、react 三种方法
└── ReAct-master/         # ReAct 原始实现（已迁移到 timeqa_baseline_lab）
```

## BGE-M3 模型说明

所有使用检索的方法（MRAG、rag_cot、react）均使用 **BGE-M3** 作为 embedding 模型。

### BGE-M3 特点
- **模型路径**: `/workspace/models/bge-m3`
- **最大长度**: 推荐 512（不建议超过 2048）
- **输出**: 返回 dense vectors, lexical weights, colbert vectors（检索使用 dense_vecs）
- **存储**: 使用 FAISS 进行高效相似度搜索
- **批处理**: 建议 batch_size=12（根据内存调整）

### 安装依赖
```bash
# 基础依赖
pip install FlagEmbedding faiss-cpu  # 或 faiss-gpu

# MRAG 额外依赖
cd /workspace/ETE-Graph/agent/MRAG-master
# 根据实际需要安装依赖（vLLM、OpenAI 等）

# timeqa_baseline_lab 依赖
cd /workspace/ETE-Graph/agent/timeqa_baseline_lab
pip install -r requirements.txt
```

### 设备选择
- **CPU**: 使用 `use_fp16=False, device='cpu'`
- **GPU**: 使用 `use_fp16=True, device='cuda'`

## 方法一：MRAG（Modular Retrieval-Augmented Generation）

### 简介
MRAG 是一个模块化的时序问答检索框架，使用原始代码实现。现已集成 **BGE-M3** 作为第一阶段检索选项。

### 位置
[MRAG-master](MRAG-master/)

### 运行步骤

#### 1. 环境准备
```bash
cd /workspace/ETE-Graph/agent/MRAG-master
```

#### 2. 生成 BGE-M3 检索结果（首次运行）
```bash
python retrieve_bgem3.py
```

**配置项**（在脚本中修改）：
- `data_path`: 问题数据路径
- `corpus_path`: Wikipedia 语料库路径
- `output_path`: 输出结果路径（如 `./TempRAGEval/BGEM3_output/TempRAGEval.json`）
- `index_dir`: FAISS 索引保存目录
- `device`: "cpu" 或 "cuda"
- `use_fp16`: 是否使用 FP16（仅 GPU）

#### 3. 设置 CUDA 设备
在 [reader.py](MRAG-master/reader.py) 中配置 GPU：
```python
os.environ["CUDA_VISIBLE_DEVICES"] = "0,1"  # 根据实际GPU情况修改
```

#### 4. 运行命令（使用 BGE-M3）
```bash
python metriever.py \
    --stage1-model bgem3 \
    --bgem3-output ./TempRAGEval/BGEM3_output/TempRAGEval.json \
    --stage2-model metriever \
    --metriever-model nv2 \
    --llm llama_8b
```

**或使用传统方法**：
```bash
python reader.py \
    --max-examples 100 \
    --retriever-output "situatedqa_bgem3_metriever_bgegemma_llama_8b_qfs5_outputs.json" \
    --ctx-topk 3 \
    --stage1-model bgem3 \
    --reader gpt \
    --paradigm concat
```

#### 5. 主要参数说明
| 参数 | 说明 | 可选值 |
|------|------|--------|
| `--max-examples` | 最大样本数量 | 整数 |
| `--retriever-output` | 检索器输出文件名 | JSON 文件名 |
| `--ctx-topk` | 使用的上下文数量 | 整数（默认：3） |
| `--stage1-model` | 第一阶段检索模型 | `bm25`, `contriever`, `bgem3`, `hybrid` |
| `--bgem3-output` | BGE-M3 检索结果文件（当 stage1-model=bgem3 时） | JSON 文件路径 |
| `--reader` | 阅读器模型 | `llama`, `llama_8b`, `llama_70b`, `gpt` |
| `--paradigm` | 阅读范式 | `fusion`, `concat` |
| `--param-pred` | 是否启用参数预测 | 布尔值（默认：True） |
| `--param-cot` | 是否启用 CoT 参数预测 | 布尔值（默认：False） |

#### 6. 输出位置
- 结果文件保存在 `./answered/` 目录
- 文件名格式：`{retriever_name}_top{ctx_topk}_{llm_name}_{paradigm}_results.csv`

#### 7. 相关文件
- BGE-M3 检索脚本: [retrieve_bgem3.py](MRAG-master/retrieve_bgem3.py)
- Metriever 主脚本: [metriever.py](MRAG-master/metriever.py)

## 方法二：Zero-Shot Chain-of-Thought (zero_shot_cot)

### 简介
零样本思维链方法，直接让 LLM 逐步推理回答问题，不使用检索到的上下文。

### 位置
[timeqa_baseline_lab](timeqa_baseline_lab/)

### 运行步骤

#### 1. 环境准备
```bash
cd /workspace/ETE-Graph/agent/timeqa_baseline_lab
```

#### 2. 配置文件修改
编辑 [configs/default.yaml](timeqa_baseline_lab/configs/default.yaml)，修改以下关键配置：

```yaml
data:
  corpus_path: "/path/to/corpus/test.json"           # 修改为实际语料库路径
  question_arrow_path: "/path/to/test/data.arrow"    # 修改为实际问题数据路径

model:
  provider: "local"                                   # 或 "remote"
  model_name: "Qwen/Qwen3-4B-Instruct-2507"          # 本地模型名称

run:
  strategy: "zero_shot_cot"                          # 设置策略为 zero_shot_cot
  max_questions: 100                                 # 处理的问题数量
```

#### 3. 运行命令
```bash
# 使用默认配置
python -m timeqa_baseline_lab.run

# 或者使用命令行覆盖参数
python -m timeqa_baseline_lab.run \
    --config configs/default.yaml \
    --strategy zero_shot_cot \
    --max_questions 100
```

#### 4. 方法特点
- **无需检索**：直接使用 LLM 的参数化知识
- **思维链提示**：要求模型逐步思考后给出答案
- **提示格式**：`"Question: {question}\nThink step by step silently, then output only:\nFinal Answer: <answer>"`

## 方法四：ReAct (Reasoning + Acting)

### 简介
ReAct 方法通过交替执行推理（Thought）和行动（Action）来解决问题，支持多步检索和推理。使用 **BGE-M3** 作为检索模型。

### 位置
[timeqa_baseline_lab](timeqa_baseline_lab/)（代码已从 ReAct-master 迁移）

### 运行步骤

#### 1. 环境准备
```bash
cd /workspace/ETE-Graph/agent/timeqa_baseline_lab
```

#### 2. 配置文件修改
编辑 [configs/default.yaml](timeqa_baseline_lab/configs/default.yaml)：

```yaml
data:
  corpus_path: "/path/to/corpus/test.json"
  question_arrow_path: "/path/to/test/data.arrow"

model:
  provider: "local"
  model_name: "Qwen/Qwen3-4B-Instruct-2507"

retriever:
  type: "bgem3"                                      # 使用 BGE-M3
  model_path: "/workspace/models/bge-m3"             # BGE-M3 模型路径
  top_k: 5
  use_fp16: false                                     # CPU 使用 false，GPU 使用 true
  device: "auto"                                      # 可选: "cpu", "cuda", "auto"

run:
  strategy: "react"                                  # 设置策略为 react
  max_questions: 100
  strategy_params:
    max_steps: 6                                     # ReAct 最大步数
```

#### 3. 运行命令
```bash
# 使用默认配置
python -m timeqa_baseline_lab.run

# 或者使用命令行覆盖参数
python -m timeqa_baseline_lab.run \
    --config configs/default.yaml \
    --strategy react \
    --max_questions 100
```

#### 4. 方法特点
- **交互式推理**：Thought（思考）→ Action（行动）→ Observation（观察）循环
- **支持的行动**：
  - `Search[entity]`：使用 BGE-M3 检索相关文档
  - `Lookup[keyword]`：在当前文档中查找关键词
  - `Finish[answer]`：给出最终答案
- **最大步数**：默认 6 步（可在配置中修改）
- **Few-shot 示例**：使用预定义的示例引导模型

#### 5. 相关文件
- ReAct Agent 实现: [migrated/react_agent.py](timeqa_baseline_lab/src/timeqa_baseline_lab/migrated/react_agent.py)
- BGE-M3 检索器: [retriever.py](timeqa_baseline_lab/src/timeqa_baseline_lab/retriever.py)

### ReAct 实现对齐报告

本节详细说明 `timeqa_baseline_lab` 中的 ReAct 实现与原始 `ReAct-master` 的对齐情况。

#### 修改目标

使 `timeqa_baseline_lab` 中的 ReAct 方法与 `ReAct-master` 中的原始实现**完全对齐**（除数据源外）。

#### 核心修改

##### 1. 移除 Fallback 机制

**修改位置**: [react_agent.py:145-151](timeqa_baseline_lab/src/timeqa_baseline_lab/migrated/react_agent.py#L145-L151)

**修改前**:
```python
if not answer:
    final_prompt = (
        f"Question: {question}\n"
        f"Evidence:\n{_format_evidence(collected, top_k)}\n\n"
        "Output only: Final Answer: <answer>"
    )
    answer = llm.generate(final_prompt, system_prompt=SYSTEM_PROMPT)

return answer, collected[:top_k], trace
```

**修改后**:
```python
# IMPORTANT: No fallback mechanism (aligned with ReAct-master behavior)
# If the agent fails to call Finish[] within max_steps, answer remains empty
# This is the original ReAct behavior - failed reasoning = no answer

return answer, collected[:top_k], trace
```

**影响**:
- ✅ 与原版行为一致：如果在 `max_steps` 内未调用 `Finish[]`，返回空答案
- ⚠️ 可能降低答案覆盖率（约 5-15%），但符合原版 ReAct 的严格推理要求

##### 2. 增强文档说明

**修改位置**: [react_agent.py:75-102](timeqa_baseline_lab/src/timeqa_baseline_lab/migrated/react_agent.py#L75-L102)

新增的文档说明明确了：
- 实现忠实复制了原始 ReAct 算法
- 与原版的差异仅限于数据源（本地 retriever vs Wikipedia API）
- 行为对齐的关键点（无 fallback、相同的循环结构、相同的动作语义）

##### 3. 修复 BOM 问题

**修改位置**: [__init__.py:1](timeqa_baseline_lab/src/timeqa_baseline_lab/migrated/__init__.py#L1)

移除了文件开头的 BOM (Byte Order Mark) 字符 `\ufeff`，该字符会导致 Python 解析错误。

#### 对齐验证

##### 与 ReAct-master 的对比

| 特性 | ReAct-master | timeqa_baseline_lab (修改后) | 对齐状态 |
|------|--------------|------------------------------|---------|
| **循环结构** | `for i in range(1, 8)` | `for i in range(1, max_steps + 1)` | ✅ 一致 (默认6步) |
| **Prompt 构建** | instruction + examples + question | instruction + examples + question | ✅ 完全一致 |
| **Search 行为** | Wikipedia API 搜索 + 前5句 | retriever.search() + 前5句 | ✅ 逻辑一致，数据源不同 |
| **Lookup 行为** | 在当前页面查找包含 keyword 的句子 | 在当前页面查找包含 keyword 的句子 | ✅ 完全一致 |
| **Finish 行为** | 提取答案并终止 | 提取答案并终止 | ✅ 完全一致 |
| **Fallback 机制** | ❌ 无 | ❌ 无（已移除） | ✅ 对齐 |
| **失败处理** | 返回空答案 | 返回空答案 | ✅ 对齐 |
| **证据收集** | ❌ 不返回 | ✅ 返回（用于评估） | ⚠️ 额外功能（不影响核心行为） |
| **执行轨迹** | ❌ 不返回 | ✅ 返回（用于调试） | ⚠️ 额外功能（不影响核心行为） |

#### 保留的合理差异

以下差异是有意保留的，因为它们**不影响 ReAct 核心推理行为**，只是增强了可观测性和评估能力：

##### 1. 证据收集 (Evidence Collection)

```python
collected.extend(found)  # 每次 Search 时收集文档
return answer, collected[:top_k], trace  # 返回收集的文档
```

**用途**:
- 评估检索质量（Recall@k, Precision）
- 分析答案来源
- 验证推理过程的合理性

**不影响推理**: 仅在返回值中添加，不参与 LLM 推理过程

##### 2. 执行轨迹 (Execution Trace)

```python
trace.append(step_str)  # 记录每一步
return answer, collected[:top_k], trace  # 返回轨迹
```

**用途**:
- 调试 ReAct 推理过程
- 分析失败原因
- 生成案例研究

**不影响推理**: 仅用于记录，不参与决策

##### 3. 块去重 (Chunk Deduplication)

```python
dedup = {}
for c in collected:
    dedup[c.chunk_id] = c
collected = list(dedup.values())
```

**用途**: 避免重复检索相同文档

**不影响推理**: 仅优化返回的证据列表，不影响 LLM 看到的 Observations

| `temperature` | 生成温度（0.0 表示贪婪解码） | 所有方法 |
| `max_new_tokens` | 最大生成 token 数 | 所有方法 |

### 评估指标

所有方法都会输出以下评估指标：
- **Accuracy**：精确匹配准确率
- **F1 Score**：Token 级别的 F1 分数
- **分数据集评估**：TimeQA 和 SituatedQA 分别评估

## 故障排除

### 常见问题

1. **CUDA 内存不足**
   - 减少 `batch_size`
   - 使用更小的模型
   - 减少 `max_new_tokens`
   - BGE-M3: 设置 `use_fp16=False` 或切换到 CPU

2. **找不到数据文件**
   - 检查 `configs/default.yaml` 中的路径配置
   - 确保路径使用绝对路径或正确的相对路径

3. **模型加载失败**
   - 检查 HuggingFace 模型名称是否正确
   - 确保有足够的磁盘空间下载模型
   - 检查网络连接（首次下载时）
   - BGE-M3: 确认模型路径 `/workspace/models/bge-m3` 存在

4. **API 调用失败**
   - 检查 API key 环境变量是否设置
   - 验证 `base_url` 配置是否正确
   - 检查网络连接

5. **BGE-M3 相关问题**
   - **FlagEmbedding 导入错误**: 运行 `pip install FlagEmbedding`
   - **FAISS 导入错误**: 运行 `pip install faiss-cpu` 或 `faiss-gpu`
   - **Bus Error**: 降低 `max_length`（不要超过 2048）
   - **索引构建缓慢**: 首次运行正常，后续会使用缓存
   - **内存不足**: 切换到 CPU 模式或减少 batch_size

6. **检索器类型错误**
   - 确保配置文件中 `retriever.type` 设置为 `"bgem3"`
   - 如果使用 MRAG，确保运行了 `retrieve_bgem3.py`

## BGE-M3 注意事项

### 性能优化
1. **内存使用**: BGE-M3 模型较大，建议在有足够内存的机器上运行
2. **max_length**: 不建议超过 2048，可能导致 bus error
3. **批处理大小**: 建议 batch_size=12（根据内存调整）
4. **缓存机制**: 两个实现都支持缓存 embeddings，避免重复计算

### 设备选择
- **CPU 模式**:
  ```yaml
  use_fp16: false
  device: "cpu"
  ```
- **GPU 模式**:
  ```yaml
  use_fp16: true
  device: "cuda"
  ```
- **自动选择**:
  ```yaml
  device: "auto"  # 自动检测可用设备
  ```

### FAISS 索引
- BGE-M3 使用 FAISS 进行高效相似度搜索
- 索引会自动保存到缓存目录，避免重复构建
- 首次运行会较慢（需要构建索引），后续运行会快速加载

### 替代方案
如果 BGE-M3 不可用或资源受限，可以使用 Contriever：
```yaml
retriever:
  type: "contriever"
  model_name: "facebook/contriever"
```
